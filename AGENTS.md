# Enterprise Approval SDD Agents Guide

本文件定义企业审批管理系统项目中 Codex/Agent 的协作规则、技术偏好和交付边界。后续所有需求分析、技术方案、代码生成、测试设计和文档更新都应优先遵循本文件。

## 1. 项目背景

本项目是一个基于 SDD（Specification-Driven Development，规范驱动开发）的企业审批管理系统。

当前核心规格文件：

- `specs/001-enterprise-approval/spec.md`：产品需求规格说明。
- `specs/001-enterprise-approval/plan.md`：技术方案与实施计划。
- `specs/001-enterprise-approval/tasks.md`：任务拆分清单。

任何实现、接口、页面、测试和重构都必须能追踪到当前迭代目录下的 `spec.md` 或 `plan.md` 中的明确依据。

## 2. 项目目标

第一版系统面向几十人规模的小公司，提供 Web 端企业内部审批能力。

第一版审批类型包括：

- 请假审批
- 报销审批
- 采购审批
- 加班审批
- 出差审批

第一版重点解决：

- 审批慢
- 流程不透明
- 审批记录难追溯
- 权限边界不清
- 管理者难以查看审批进度和历史

## 3. 技术方向

### 3.1 后端

后端优先采用：

- Java
- Java 1.8
- Spring Boot 2.7.x
- Spring Security
- Spring Data JPA 或 MyBatis-Plus
- PostgreSQL

选择 Spring Boot 的原因：

- 企业管理系统与 Java/Spring 生态契合度高。
- Spring Security 适合实现登录、角色权限、接口权限和方法级授权。
- Spring 事务模型成熟，适合审批流、状态流转、审计日志等强一致业务。
- Java 类型系统稳定，适合长期维护复杂业务规则。
- Spring Boot 在企业内部部署、运维、监控、日志、集成方面更常见。

如果后续没有特别说明，后端方案应以 Spring Boot 2.7.x 为默认方向。

所有 Java 代码必须兼容 Java 1.8。不得使用 Java 9 及以上版本才支持的语言特性、标准库 API 或构建配置。

### 3.2 前端

前端优先采用：

- React
- TypeScript
- Vite
- React Router
- TanStack Query
- Ant Design

前端应定位为企业后台管理系统，不做营销页，不做花哨视觉，优先保证清晰、稳定、高效。

### 3.3 数据库

数据库优先采用：

- PostgreSQL

审批申请、流程实例、审批任务、通知、审计日志等核心数据应结构化建模。审批表单差异字段可以使用 JSON 字段，但常用查询字段应冗余为独立列。

## 4. 需求 database

数据库相关工作必须遵守以下约束：

- 禁止生成 `DROP TABLE` 语句。
- 禁止生成 `DROP DATABASE` 语句。
- 所有 SQL 必须使用参数化查询，禁止拼接用户输入生成 SQL。
- 执行数据库操作前必须验证当前环境是生产、测试还是开发。
- 危险操作必须添加二次确认机制。
- 数据不得物理删除，必须使用软删除。
- 所有业务表必须设计软删除字段，例如 `deleted`、`deleted_at`、`deleted_by`，具体字段命名在数据库设计阶段统一。
- 查询默认必须过滤已软删除数据，只有审计、恢复、管理员排查等明确场景可以查看软删除数据。
- 数据库物理表名按业务域使用前缀：`sys_`、`org_`、`approval_`、`audit_`。

危险操作包括但不限于：

- 批量删除数据。
- 批量更新关键业务状态。
- 清空表数据。
- 修改用户、角色、权限、审批流程等核心配置。
- 修改已通过、已作废、审计日志等关键历史数据。

即使在开发环境，也不得为了省事生成破坏性数据库语句。需要重建结构时，应优先通过安全迁移、备份、显式确认和可回滚方案处理。

表命名前缀规则：

- `sys_`：系统、账号、角色、权限相关表，例如 `sys_user`、`sys_role`。
- `org_`：组织架构相关表，例如 `org_department`、`org_position`。
- `approval_`：审批业务相关表，例如 `approval_request`、`approval_task`。
- `audit_`：审计相关表，例如 `audit_log`。

不得再使用统一 `user_` 或统一 `order_` 作为本项目全部数据表前缀。

## 5. 工作方式

### 5.1 SDD 优先

所有开发工作必须遵循以下顺序：

1. 先确认或更新规格。
2. 再更新技术方案。
3. 再拆分任务。
4. 再实现代码。
5. 最后补充测试和验收说明。

不得在没有规格依据的情况下直接扩大功能范围。

### 5.2 强追踪

新增功能时，必须说明对应来源：

- 来源于当前迭代 `spec.md` 的哪个章节。
- 来源于当前迭代 `plan.md` 的哪个章节。
- 如果规格缺失，应先补充规格再实现。

### 5.3 小步提交

实现时应按可验证的小步骤推进：

- 领域模型
- 数据库迁移
- 服务层
- 接口层
- 前端页面
- 权限控制
- 测试

每一步都应能独立检查。

## 6. 架构原则

### 6.0 请求与响应日志

所有后端服务/API 必须记录请求日志和响应日志。

日志要求：

- 记录请求时间、请求方法、请求路径、请求参数摘要、当前用户、客户端 IP、请求 ID。
- 记录响应时间、响应状态、耗时、错误码、错误摘要。
- 不得记录密码、令牌、收款账户完整信息、附件内容等敏感数据。
- 请求日志与响应日志必须能通过同一个请求 ID 关联。
- 异常响应必须记录异常类型和错误摘要，但不得向前端暴露内部堆栈。
- 如果后续拆分微服务，每个微服务都必须遵守同样的请求/响应日志规范。

### 6.1 后端分层

Spring Boot 后端建议采用以下分层：

- Controller：处理 HTTP 请求和响应。
- Application Service：编排业务用例。
- Domain Service：承载审批流、状态机、权限判断等核心业务规则。
- Repository/Mapper：数据访问。
- DTO：接口输入输出模型。
- Entity：数据库实体。
- Security：认证与授权。

审批状态流转、流程推进、通知、审计不应散落在 Controller 中。

### 6.2 模块边界

后端建议按业务模块组织：

- auth：认证
- user：用户
- organization：组织架构
- rbac：角色权限
- approval：审批申请
- workflow：审批流程
- notification：通知
- audit：审计日志
- dashboard：工作台和统计
- export：导出

### 6.3 核心业务规则集中管理

以下规则必须集中实现，避免散落：

- 审批状态机
- 默认审批流程
- 流程配置解析
- 审批人解析
- 数据权限判断
- 可执行动作判断
- 超时判断
- 审计写入
- 通知创建

## 7. 权限要求

系统必须同时支持：

- 菜单权限
- 操作权限
- 数据权限

前端隐藏按钮只是体验优化，不是安全边界。所有权限必须以后端判断为准。

任何审批详情、审批动作、导出、审计日志查询都必须进行服务端权限校验。

## 8. 测试要求

核心业务必须有测试覆盖：

- 审批状态机
- 表单校验
- 默认流程解析
- 金额阈值规则
- 审批动作
- 数据权限
- 通知创建
- 审计日志写入
- 超时提醒

对于审批流这种状态密集业务，不能只依赖前端手工验证。

## 9. 第一版边界

第一版明确不做：

- 移动端 App
- 短信通知
- 邮件通知
- 企业微信集成
- 钉钉集成
- BPMN 流程设计器
- 电子签章
- 发票 OCR
- 财务付款系统
- 考勤系统
- 薪资系统
- 多公司/集团架构
- 多语言
- 合同审批
- 用章审批

如需新增这些能力，必须先更新当前迭代的 `spec.md` 和 `plan.md`。

## 10. 文档维护规则

当技术方向、接口、模块、数据库模型或流程规则发生变化时，必须同步维护：

- `specs/001-enterprise-approval/spec.md`：需求变化时更新。
- `specs/001-enterprise-approval/plan.md`：技术方案变化时更新。
- `specs/001-enterprise-approval/tasks.md`：任务拆分变化时更新。
- `AGENTS.md`：项目协作规则或技术偏好变化时更新。

当前已决定：后端默认采用 Spring Boot 2.7.x + Java 1.8。当前迭代 `plan.md` 已同步为 Spring Boot 后端方案。
